# SPEC.md — マクロ経済ダッシュボード「Macro Cockpit」

Claude Code で実装するための仕様書。リポジトリ直下に置き、フェーズ単位で実装を進めること。

---

## 1. 目的

日本と世界のマクロ経済を一画面で監視する個人用ダッシュボード。
単なる経済ニュースの一覧ではなく、**運用の前提シナリオ(スタグフレーション/財政ドミナンス/円希薄化)が崩れていないかを毎朝1分で確認できる計器盤**とする。

- 利用者: 本人のみ(ローカル or 自宅サーバで動作)
- 更新頻度: 1日1回のバッチ取得で十分(リアルタイム不要)
- 成功基準: ブラウザを開いて30秒以内に「シナリオに変化なし/要注意」が判断できること

## 2. スコープ

### MVP(Phase 0–1)でやること
- FRED API と Yahoo Finance(yfinance)のみをデータソースとする(APIキーはFREDの無料キー1つだけ)
- 日次バッチでSQLiteに時系列を蓄積し、Next.jsのダッシュボードで表示

### MVPでやらないこと(Phase 2以降)
- e-Stat / 日銀時系列 / 財務省などの日本公式ソース接続
- シナリオ整合の自動判定・アラート通知
- 認証、マルチユーザー、外部公開

**注意: スコープ外の機能を実装中に思いついても追加しない。`IDEAS.md` に1行メモして先へ進むこと。**

## 3. 技術スタック

| 層 | 技術 | 理由 |
|---|---|---|
| データ取得 | Python 3.12 / pandas / requests / yfinance(Phase 2でIB API追加) | 取得・整形はpandasが最速。既存のPython資産と親和 |
| 保存 | SQLite(単一ファイル `data/macro.sqlite`) | 依存ゼロ・バックアップはファイルコピー。既存の取引システム(TimescaleDB)とは意図的に分離し、干渉させない |
| フロント | Next.js(App Router)/ TypeScript / Tailwind CSS / Recharts | サーバ側で better-sqlite3 により直接読む。APIサーバ不要 |
| 実行 | Makefile + cron(または手動 `make fetch`) | 常駐プロセスなし |

### 4.2 Phase 2で追加する日本公式ソース
- e-Stat API(appId要): 消費者物価指数の詳細、毎月勤労統計(実質賃金)
- 日銀 時系列統計データ: マネタリーベース、日銀総資産、無担保コールO/N(政策金利)(CSVダウンロード)
- 国土交通省 不動産価格指数: 住宅・区分マンション(月次、約2〜3ヶ月ラグ、Excel/CSV)
- 財務省 国債金利情報: JGB各年限の日次利回り(CSV)。特に**30年**は財政ドミナンス懸念のセンサーとしてパネル化する
- 財務省 貿易統計: 貿易収支(月次)— 円の実需フローの構造監視
- FRED: 米連邦政府利払い費(四半期、系列ID要確認)— 財政ドミナンスの直接監視
- BIS: 円の実質実効為替レート

### 4.3 取得層の設計方針
- ソースごとに `fetcher/src/fetcher/sources/{fred,yahoo}.py` として分離し、共通インターフェース(`fetch(series) -> DataFrame[date, value]`)に揃える
- yfinance は非公式APIで壊れやすい。取得層を抽象化し、将来 stooq 等へ差し替え可能にしておく
- 各ソースの利用規約の範囲(個人利用・再配布なし)で使用する

### 4.4 IB Gateway接続(Phase 2で先物・FXの一次ソース化)

実行環境では取引システム用のIB Gatewayが常時稼働しているため、先物・FX・指数の一次ソースとしてIB APIを使える。yfinanceの先物データは非公式で不安定なため、**Phase 2で先物・FX系列の一次ソースをIBに切り替え、yfinanceはフォールバックに降格する。**

- `sources/ib.py` を追加。既存の取引システムで使用中のIBクライアント資産(接続・日足バー取得部分)を流用してよい
- 先物(GC/CL/HG)は ContFuture で日足を取得、USDJPY は IDEALPRO のミッドを使用
- 取得後は即切断する**ワンショット接続**とし、常駐しない

**本番トレーディングシステムと同居するための必須ガードレール:**
- 取引システムと重複しない専用の clientId を使う
- `ib.py` には発注系のコードを一切書かない(データ取得関数のみをimport)
- cron の実行時刻は取引システムの稼働に影響しない時間帯(例: 早朝のメンテ時間帯)に設定
- 接続失敗・pacing制限時はリトライで帯域を占有せず、即座に yfinance フォールバックへ切り替える
- 市場データ購読がない銘柄は、遅延データ設定(reqMarketDataType)での履歴取得可否を実装時に確認し、取れない系列はフォールバック側を正とする

MVPではIBに接続しない。ダッシュボードの骨格が安定してから、Phase 2の最初のタスクとして切り替えること(本番Gatewayに触るのは骨格の検証後、という順序を守る)。

## 5. データベース設計

```sql
CREATE TABLE series (
  series_id TEXT PRIMARY KEY,   -- 例: 'FRED:DGS10', 'YF:GC=F'
  source    TEXT NOT NULL,      -- 'fred' | 'yahoo'
  name_ja   TEXT NOT NULL,      -- 表示名(日本語)
  unit      TEXT,               -- '%', '円', 'USD' など
  freq      TEXT NOT NULL       -- 'D' | 'W' | 'M'
);

CREATE TABLE observations (
  series_id TEXT NOT NULL REFERENCES series(series_id),
  date      TEXT NOT NULL,      -- ISO 8601 (YYYY-MM-DD)
  value     REAL,
  PRIMARY KEY (series_id, date)
);

CREATE TABLE fetch_log (
  ts        TEXT NOT NULL,
  series_id TEXT NOT NULL,
  status    TEXT NOT NULL,      -- 'ok' | 'error'
  message   TEXT
);
```

- 書き込みは **UPSERT(INSERT OR REPLACE)で冪等** にする。`make fetch` を何度実行しても重複・破損しないこと
- 1ソースの取得失敗は警告ログにとどめ、他ソースの取得は継続する
- 前年比(YoY)などの派生値はDBに保存せず、表示層で計算する

### 6.2 パネル共通仕様
- 上段: 指標名(日本語)+ 最新値(等幅・tabular-nums で大きく)
- 中段: 直近2年のラインチャート(Recharts、軸ラベル最小限)
- 下段: 前月比 / 前年比を小さく表示(上昇=エメラルド、下降=ローズの国際式)
- データ欠損時はパネル内に「取得待ち」と表示し、**ページ全体は絶対に落とさない**

### 6.3 シグネチャ要素: レジーム・ストリップ
画面最上部に、シナリオ監視対象のコア指標(日米CPI、JGB10年、USD/JPY、金、日米中銀B/S、VIX)を1指標=1セルの横帯で並べる。
- MVPでは「直近値が3ヶ月前より上か下か」で色分けするだけの静的表示
- Phase 3でシナリオ整合ルール(例: 円安×金高×実質金利低下=シナリオ継続)を実装する
- このストリップだけ見れば毎朝の確認が終わる、というのがこの画面の存在意義

### 6.4 デザイン指針(Grafana純正風ダーク・A案採用)
- ダークテーマ固定。計器盤(コックピット)の質実さを狙い、装飾やアニメーションは足さない(Rechartsは `isAnimationActive={false}`)
- 配色トークン: 背景 `#111217` / パネル `#181B1F` / 罫線 `#2C3235` / 本文 `#F5F6F8` / サブテキスト `#9DA5B8` / 主系列(緑)`#73BF69` / 第2系列(金利・リスク系、青)`#5794F2` / 警告(黄)`#F2CC0C` / 上昇 `#73BF69` / 下降 `#FF7383`
- レジーム・ストリップの色はGrafanaの閾値色の流儀に合わせる: シナリオ整合=緑 `#73BF69` / 中立=グレー `#2C3235` / 逆行=赤 `#FF7383`
- パネルの型: 左上にタイトル(11px・サブテキスト色)、その下に最新値(20〜24px・等幅・tabular-nums)と騰落率(右寄せ・上昇/下降色)、下部にチャート。パネルは角丸8px+ヘアライン罫線
- チャート: ライン1.5px+同色・不透明度12%のフラットな面塗り(グラデーション不可)。軸・目盛りは最小限とし、グリッド線は罫線色の点線。ツールチップはホバー時のみ
- フォント: 本家Grafanaに合わせ、数値・ティッカーは Roboto Mono(tabular-nums 必須)、日本語UIは Noto Sans JP、欧文UIは Inter
- レスポンシブはデスクトップ優先。モバイルは1カラム縦積みで崩れなければ十分

## 8. 開発フェーズと受け入れ基準

### Phase 0: 骨格(半日)
- リポジトリ雛形、Makefile、DBスキーマ初期化、FREDから `DGS10` 1系列のみ取得
- **受け入れ基準:** `make fetch` でDGS10がSQLiteに保存され、再実行しても行数が増えない。`fetch_log` に記録が残る

### Phase 1: MVP(1〜2日)
- §4.1の全系列の取得、Next.jsダッシュボード(16パネル+レジーム・ストリップ静的版)
- **受け入れ基準:**
  - `make fetch && make dev` で全パネルが表示される
  - 任意の1ソースを意図的に失敗させてもページが表示される(該当パネルのみ「取得待ち」)
  - 要確認の系列IDがすべて検証済みで、確定版の系列表がREADMEにある

### Phase 2: IB接続と日本公式データ(週末2回分)
- 先物・FX系列の一次ソースをIB Gateway(§4.4)へ切り替え(yfinanceはフォールバックへ降格)
- e-Stat / 日銀CSV / 財務省CSVの取得層追加。JGB10年の日次化と30年の追加、実質賃金・マネタリーベース・政策金利・貿易収支のパネル追加
- **受け入れ基準:** 追加ソースが失敗してもMVP部分は無傷。IB接続失敗時は自動でyfinanceにフォールバックし、取引システム側に一切影響を与えない。系列の重複期間は公式ソース/IBを優先

### Phase 3: レジーム判定と常時運用
- レジーム・ストリップのルールエンジン化、cron設定、(任意)VPS配備と朝の通知
- **受け入れ基準:** 判定ルールがコードでなく設定(YAML等)で記述され、テストがある

## 9. Claude Code での進め方(CLAUDE.md に転記する内容)

- 実装は必ず SPEC.md のフェーズ順に進め、フェーズをまたぐ先回り実装をしない
- 各フェーズの受け入れ基準を満たしたことを確認してからコミットする(コミット単位=フェーズまたはその一部)
- `data/` 配下のファイルは削除・上書きしない(唯一の蓄積データのため)
- 系列の追加・変更は `fetcher/src/fetcher/registry.py` のみで行う(定義の単一化)
- コマンド: `make init`(初期化)/ `make fetch`(取得)/ `make dev`(開発サーバ)/ `make test`
- スコープ外の提案は実装せず IDEAS.md への追記を提案する

## 10. 非機能要件

- 秘密情報は `.env` のみ(コミット禁止)。必要なキーは FRED_API_KEY のみ(https://fred.stlouisfed.org/docs/api/api_key.html で無料取得)
- ネットワークアクセスは fetcher のみ。web は SQLite を読むだけ
- 全処理がラップトップ1台で完結すること(常駐なし、外部DBなし)
